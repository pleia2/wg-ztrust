## Email versus Code Signing

Berry was able to load the `.asc` bundle of PGP keys to one utility
(Kleopatra) though only the first key in the bundle was loaded to
another utility (Thunderbird with built-in OpenPGP). Based on Berry's
experience, we need to keep in mind that not all applications will
be able to consume a "bundle" of PGP keys in `.asc` armored format.

Related: The bundle that Berry imported included keys for signing
and keys for email. This made it apparent that we should distinguish
keys used for signing (whether code signing orkey signing or other)
from keys used primarily for email (signing and encryption).
Some community members will have and use just *one* PGP key,
and that's okay.

## Utilities in the PKI realm

Rick complained about not finding "one shot command" means of signing
files or documents in PKI space. There seem to be a growing number of
proprietary solutions, but nothing usable as an `openssl` command.
Most online documentation for using OpenSSL for document signing
talks about multi-step key-gen, hash, and encrypt for file signing,
which is what PGP already does as a one-shot but does not use PKI
certificates.

Berry mentioned PKI-encrypted email from a former job which he now
cannot read because he no longer has the hardware PKI token.
This immediately reminded Rick that PKI-encrypted (and signed) email
is well established and can use the standard PKI root certificate
collection. So we have a new lead in that pursuit.

Mark mentioned SMP/e package signing becoming more common
in the z/OS world and provided links to web references.
